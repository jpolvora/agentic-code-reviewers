# Auto-Fix flow (self-healing loop)

> The **Agentic Code Reviewers** supports an automated, continuous review and fix loop (Auto-Fix). The process creates a *self-healing* cycle in which code is iteratively reviewed, fixed and validated until all pending items are resolved or the round limit is hit.
>
> See [`index.md`](index.md) for an overview and [`workflows.md`](workflows.md) § Auto-fix loop for the workflow trigger details.

## How the cycle works

The Auto-Fix cycle coordinates two separate pipelines: **review** (`code-review.yml`) and **fix** (`auto-fix.yml`). The flow goes:

1. **Initial trigger:** the developer pushes code to a Pull Request.
2. **Review:** the Code Review pipeline (`Agentic Code Review`) analyzes the code. If it finds problems, it posts comments on the PR.
3. **Auto-fix trigger:** via the `workflow_run` event (in GitHub Actions), the conclusion of the review pipeline automatically triggers the Auto-fix pipeline.
4. **Fix:** the Auto-Fix agent:
   - Reads active threads (unresolved comments) on the PR.
   - Analyzes the code and proposes fixes for each problem.
   - Applies changes to the local code.
   - Generates a local commit (`fix(#N): auto-fix issues from review threads [threadId, ...]`), validates the build, resolves threads and pushes (cooperative gate — see [`../skills/COOPERATIVE_FIX.md`](../skills/COOPERATIVE_FIX.md)).
   - Posts a **detailed summary comment** on the PR listing changed files, resolved issues, and links to each discussion thread (tagged with `<!-- auto-fix-summary -->`).
5. **Re-evaluation (the loop):** the push generated by Auto-Fix re-triggers the Code Review pipeline (step 2). The reviewer evaluates the new code. If a thread was fixed, it marks it resolved. Otherwise it leaves new findings.

## Threads, score_min and final summary

- **Auto-fix processes all open review threads** on the PR (with file and line), with no tag filter. Threads that aren't a correctable code issue are ignored by the sub-agent (`replacements: []`).
- **`AGENTIC_CODE_REVIEWERS_SCORE_MIN`** defines what the **reviewer** publishes as a new thread.
- The **review summary comment** ("PR ready to merge") is posted only **at the end** of a review round, when there are **no remaining** active/pending bot threads.
- The **auto-fix summary comment** is posted after each successful auto-fix push, detailing files changed, resolved threads, and thread links — giving the developer visibility into what was fixed and triggering the next code review round.

## System protections

A continuous machine loop (evaluation and fix) raises concerns about **infinite loops** and **concurrency**. The system is designed with protections for these scenarios:

### 1. Infinite loop and convergence (round escalation)

The reviewer tracks how many review/fix rounds have occurred using an invisible comment on the PR (`<!-- reviewer-round-state -->`). If the fix fails repeatedly to solve the problems after a maximum number of attempts (configured via `AGENTIC_CODE_REVIEWERS_MAX_ROUNDS`, default: 10):

- The reviewer stops generating suggestions and warnings (lower-priority threads).
- It leaves only the **critical** issues open (if any).
- It posts a warning asking for **human review** (escalation).

This ensures the Auto-Fix pipeline won't waste CI minutes trying to fix something beyond its capacity or suffering hallucination.

### 2. Change detection (no-op)

If the Auto-Fix agent processes the threads but fails to formulate concrete code changes, the script detects a clean working tree and ends without a commit. **No commit, no resolution, no push.**

### 3. Cooperative gate (build → resolve → push)

Order aligned with the IDE skill [`solve-pr`](../.agents/skills/solve-pr/SKILL.md) and the [`COOPERATIVE_FIX.md`](../skills/COOPERATIVE_FIX.md) contract:

1. **Local** commit
2. Validation **build** (`npm test` when `package.json` has `scripts.test`, else `npm run build`, or `AGENTIC_CODE_REVIEWERS_AUTO_FIX_BUILD_COMMAND`). Failure aborts the job (exit ≠ 0).
3. Reply and resolve threads on the PR
4. **Push** only if build and resolutions succeeded

If the build or the resolution fails (token, permission, compilation), the local commit is preserved and the push is aborted.

### 4. Concurrency and sequencial execution (no overlap)

- **Per-PR `concurrency`** on the job (`auto-fix-<pr_number>`) — overlapping runs are queued.
- **Sequential engines:** cursor-sdk → conditional sync (preserves unsent local commits) → opencode. If opencode finds no active threads it tries a **recovery push** of the pending commit from the previous engine.
- **Final dual-engine gate:** fails if cursor-sdk failed, opencode ended green and local HEAD ≠ remote.
- **Partial resolution:** only threads whose line had its content changed.
- **Explicit failure** if all configured engines fail.

### 5. Trigger after manual review

The job accepts a successful `workflow_run` regardless of the original event (`pull_request` or `workflow_dispatch`).

## Required configuration (GitHub Actions)

For the loop to work, the `git push` done by the Auto-Fix script must be able to re-trigger the `pull_request` pipeline (`code-review.yml`).

> [!WARNING]
> The default GitHub Actions token (`secrets.GITHUB_TOKEN`) **intentionally prevents** invoking other workflows to avoid accidental loops. If your pipeline uses that token, the review **will not** be re-triggered and the cycle stops at the first fix (code will be adjusted, but feedback on the fix will only come when a human triggers the pipeline or pushes).

To enable the continuous loop:

1. Create a **Personal Access Token (PAT)** on GitHub, using a service account, *bot*, or your own account. The token needs read and write permissions on the repository (`repo` on *classic tokens*, or code/PR permission on *fine-grained*).
2. Add this PAT as a Secret (e.g. `AGENTIC_CODE_REVIEWERS_GITHUB_TOKEN`) in the repository settings.
3. Configure the auto-fix pipeline to pass this token to the script.

## Local auto-fix

You can run auto-fix locally with PR context and a write token:

```bash
npm run review -- --gh --pr-id 42 --source-branch feat/x --target-branch main --auto-fix
```

`--dry-run` is respected (simulates commit/resolution without a real POST).

## Failure modes

| Symptom | Likely cause |
|---------|--------------|
| Loop stops after the first fix | Default `GITHUB_TOKEN` doesn't re-trigger workflows — use PAT |
| Auto-fix produced no commit | Working tree clean (no replacements); terminates without push |
| Loop stopped before resolving | `MAX_ROUNDS` escalated to human handoff |
| Job failed at the final step | All configured engines failed; or cursor-sdk failed and opencode left a divergent local HEAD |

Full workflow details: [`workflows.md`](workflows.md) § Auto-fix loop.

## Known pitfalls

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| Loop stops after the first fix | Default `GITHUB_TOKEN` doesn't re-trigger workflows | Use a PAT as `AGENTIC_CODE_REVIEWERS_GITHUB_TOKEN` |
| Auto-fix produces no commit but threads exist | Path case mismatch resolved — upgrade to latest release which preserves original-case paths for filesystem lookups while keeping dedup case-insensitive |
| Loop stopped before resolving | `MAX_ROUNDS` escalated to human handoff | Review the escalation comment on the PR |
| `Resource not accessible by integration` | Default token lacks `resolveReviewThread` | PAT with `pull-requests: write` resolves this |
| Auto-fix skips backend files on Linux | Fixed in latest: `filePath` on threads is now canonical (original case); dedup remains case-insensitive | Pull the latest `release` branch |
